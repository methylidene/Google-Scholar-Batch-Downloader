# Google Scholar 作者主页支持设计规格

## 目标

扩展现有 Chrome Manifest V3 扩展，使用户能在 Google Scholar 作者主页选择当前已加载的论文，顺序读取论文详情以寻找明确的开放 PDF，并复用现有下载、RIS/BibTeX/JSON 导出和 Zotero 流程。

## 范围

- 支持 `https://scholar.google.com/citations?user=...` 作者主页。
- 只处理页面当前已加载的论文，不主动点击“显示更多”。
- 用户手动点击“显示更多”后，扩展自动识别新增论文。
- 不自动翻页、不绕过验证码或付费墙、不使用可疑镜像。

## 作者主页解析

作者主页解析器以 `.gsc_a_tr` 为论文行，提取标题、页面可见作者、年份、期刊或会议、详情 URL，并使用作者主页顶部姓名作为缺失作者字段的回退值。每条论文继续使用现有统一 Paper 模型。

作者主页列表通常不直接提供 PDF。选中论文执行批次时，后台顺序读取其 Scholar 详情 URL，从详情 HTML 中保守提取明确 `[PDF]` 链接、URL 路径为 `.pdf` 的 HTTP(S) 链接，以及明确 DOI 文本或 `doi.org` 链接。

## 用户界面

- 每条作者主页论文行增加复选框和状态。
- 页面顶部显示与搜索结果页一致的工具栏。
- 支持全选、取消、题目筛选、仅选择已找到 PDF 的论文、下载并导出、发送到 Zotero。
- 新增条目通过 DOM 变化监听加入，不重复插入控件。
- 状态包含待查询、已选择、查询详情、PDF、仅元数据、下载中、成功和失败。

## 数据流

1. 内容脚本识别页面类型并调用作者主页解析器。
2. 用户选择当前已加载的论文。
3. 点击“下载并导出”后，将规范化论文发送给后台。
4. 对缺少 `pdfUrl` 且具有 Scholar 详情 URL 的论文，后台按选择顺序读取详情。
5. 详情查询间隔使用 `downloadDelayMs`，限定为 300–5000 毫秒，默认 800 毫秒。
6. 找到开放 PDF/DOI 时补充 Paper；找不到时保留原元数据并标记为仅元数据。
7. 查询阶段完成或中止后，复用现有 PDF 下载与 RIS/BibTeX/JSON 生成流程。
8. Zotero 继续作为独立操作，失败时保留文件导出降级。

## 错误处理

- 详情请求严格串行，不并发访问 Scholar。
- 单篇网络或解析失败不丢失元数据，也不阻止普通单篇失败后的后续项目。
- 检测到 CAPTCHA 或异常流量页时立即停止剩余详情查询，保留已完成结果并生成导出文件。
- 详情响应不是 HTML、页面结构无效或无明确 PDF 时，不猜测下载地址。
- MutationObserver 只处理新增论文行，并通过稳定标记防止重复控件。

## 权限与隐私

无需新增 `tabs` 权限。Manifest 将 content script 匹配范围扩展到 Scholar `citations` 页面；详情读取使用现有 `https://scholar.google.com/*` 站点权限。数据仍在本地处理，不上传搜索或作者浏览记录。

## 测试

- 作者主页固定 HTML：标题、作者、年份、期刊和详情 URL。
- 作者姓名回退与缺失字段。
- 当前已加载行选择，以及 MutationObserver 处理“显示更多”新增行且不重复。
- 详情 HTML 的 PDF/DOI 提取、HTML 链接拒绝和无 PDF 情况。
- 顺序查询、300–5000 毫秒节流、单篇失败与 CAPTCHA 中止。
- 作者主页工具栏选择和状态变化。
- 保持现有搜索结果页测试全部通过。

## 完成标准

- 作者主页当前已加载论文可被识别、选择和筛选。
- 手动“显示更多”后的新论文自动获得选择控件。
- 所选论文详情按顺序、限速读取；明确开放 PDF 可下载。
- 找不到 PDF 的条目仍进入 RIS/BibTeX/JSON 和 Zotero 元数据流程。
- CAPTCHA 中止后不继续请求，并保留已完成结果。
- 搜索结果页现有行为无回归，README 包含作者主页操作说明。
