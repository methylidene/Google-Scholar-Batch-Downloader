# 下载汇报功能设计规格

## 目标与范围

为 Google Scholar Batch Downloader 增加批次结束汇报：准确区分实际下载完成、未找到 PDF、下载失败和超时，并在 Scholar 页面显示汇总面板，同时自动生成 CSV 报告。

本阶段不实现备用下载源，只在结果中预留 `source` 字段，默认值为 `scholar`。

## 成功标准

Chrome 接受下载任务并返回 `downloadId` 只代表“下载中”。只有监听到 `chrome.downloads.onChanged` 的 `state.current === 'complete'` 才标记为成功；监听到 `interrupted` 标记为失败并记录中断原因。批次等待超过四分钟仍未结束的项目标记为超时。

## 状态分类

- `success`：PDF 文件实际下载完成。
- `no_pdf`：没有找到可下载 PDF，仅保留元数据。
- `failed`：PDF 下载启动失败，或启动后被 Chrome 中断。
- `timeout`：四分钟内没有收到完成或中断状态。

RIS、BibTeX、JSON 和 CSV 等报告文件的下载错误单独记录，不混入论文 PDF 状态。

## 数据结构

每篇论文的最终结果包含：

- `id`
- `title`
- `authors`
- `year`
- `status`
- `source`，首版固定为 `scholar`
- `pdfUrl`
- `filename`
- `downloadId`
- `error`
- `startedAt`
- `finishedAt`

## 后台流程

1. 对没有 PDF URL 的条目立即生成 `no_pdf` 结果。
2. 对有 PDF URL 的条目按现有规则启动下载，启动失败最多重试一次。
3. 启动成功后保存 `downloadId`，监听下载事件。
4. 多篇论文允许乱序结束；每个事件按 `downloadId` 更新对应条目。
5. 全部条目进入终态，或四分钟批次等待超时后，生成 RIS、BibTeX、JSON 和 CSV。
6. CSV 下载失败时，后台响应保留完整结果，并记录报告错误。
7. 备用下载源留到后续版本；首版不会在失败后自动访问其他站点。

## CSV 报告

CSV 使用 UTF-8 BOM，字段为：标题、作者、年份、最终状态、来源、PDF URL、文件名、downloadId、失败原因、开始时间、结束时间。

所有单元格按 RFC 4180 兼容方式转义：包含逗号、双引号、回车或换行时使用双引号包围，内部双引号写成两个双引号。

## 页面汇总面板

新批次开始时移除旧汇报。批次结束后在 Scholar 页面显示：

- 总数
- 成功下载数
- 未找到 PDF 数
- 下载失败数
- 超时数
- 导出/报告文件错误
- 可展开的逐篇明细
- “关闭汇报”按钮

逐篇明细显示标题、状态、文件名、下载来源和失败原因。所有动态文本使用 `textContent` 创建，不把论文标题、URL 或错误信息拼接到 `innerHTML`。

## 容错与生命周期

- 单篇失败不影响其他条目。
- 页面关闭或刷新后不恢复未完成汇报，首版不做跨会话持久化。
- 四分钟超时是 Manifest V3 后台生命周期保护，超时结果不会自动重试。
- 页面汇总无法显示时，CSV/JSON 仍保留最终结果。
- 报告文件下载失败时，页面汇总显示明确错误。

## 测试

- 下载 `complete`、`interrupted` 和超时事件。
- 多篇下载乱序完成与事件按 ID 关联。
- 启动失败重试一次，且不影响其他条目。
- `no_pdf` 分类。
- CSV 中文、逗号、双引号、换行、UTF-8 BOM。
- 页面汇总计数、逐篇明细、关闭按钮和安全 DOM 写入。
- 报告下载失败时的页面提示。
- 搜索结果页、作者主页、详情补全、导出和 Zotero 现有测试全部保持通过。

## 完成标准

- 成功状态只在实际下载完成后产生。
- 所选论文全部出现在页面汇总和 CSV 中。
- 成功、未找到 PDF、失败和超时计数准确。
- 失败原因与 downloadId 可追踪。
- CSV 可在 Windows Excel 中正确显示中文。
- 现有搜索结果页与作者主页功能无回归。
